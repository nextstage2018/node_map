CONTEXT FLOW ARCHITECTURE
================================================================================

SECRETARY CHAT (Entry Point)
  User Message
    ↓
  Intent Classification (keyword-based, NO API)
    ↓
  Parallel DB Fetches (Promise.all)
    ├─ inbox_messages (20 recent, received)
    ├─ tasks (20 recent, status != done)
    ├─ jobs (15 recent)
    ├─ calendar_events (if briefing intent)
    └─ Special tables (consultations, file_staging, proposals, etc)
    ↓
  Format Context Text (3000-5000 chars)
    ├─ Unread count + urgency badges
    ├─ Task statuses (proposed/in_progress/completed)
    ├─ Job approval counts
    ├─ Calendar preview (if connected)
    └─ File/knowledge/consultation counters
    ↓
  Claude API Call
    ├─ Model: claude-sonnet-4-5-20250929
    ├─ Max tokens: 2000
    ├─ System: Intent-specific instructions
    └─ Messages: User input + formatted context
    ↓
  Card Generation
    ├─ BriefingSummaryCard (statistics dashboard)
    ├─ CalendarCard (today's events)
    ├─ InboxSummaryCard (recent messages)
    ├─ TaskResumeCard (task proposals)
    ├─ JobApprovalCard (pending jobs)
    ├─ FileIntakeCard (pending files)
    ├─ KnowledgeProposalCard (pending clusters)
    └─ [15+ other card types]
    ↓
  Return Response + Cards to UI

TASK AI CHAT (Conversation)
  User Message in Task Detail
    ↓
  Load Task Context
    ├─ task.title, description, ideation_summary
    ├─ project.name, description
    ├─ organization.name, memo
    ├─ task_members → contact_persons names
    └─ task_conversations (last 20 messages)
    ↓
  Detect Covered Items (Regex)
    ├─ Has "ゴール" discussed? → true/false
    ├─ Has "内容" discussed? → true/false
    ├─ Has "懸念" discussed? → true/false
    └─ Has "期限" discussed? → true/false
    ↓
  Format System Prompt
    ├─ Phase-specific: Ideation (4-item guided) / Progress (support) / Result (summary)
    ├─ Task info section
    ├─ Project context section (if available)
    ├─ Covered items section (prevents repetition)
    └─ Writing style sample (if available)
    ↓
  Claude API Call
    ├─ Model: claude-sonnet-4-5-20250929
    ├─ Max tokens: 1500
    ├─ System: Full system prompt (1500+ chars)
    └─ Messages: Last 20 conversations + user message
    ↓
  Save Conversation Turn
    ├─ User message → task_conversations
    ├─ AI response → task_conversations
    ├─ turn_id (UUID) → for conversation jumping
    └─ phase (ideation/progress/result)
    ↓
  Background: Keyword Extraction (Async)
    ├─ Text: user message + AI response
    ├─ Service: ThoughtNodeService.extractAndLink()
    ├─ Sub-call: keywordExtractor.service.ts
    ├─ Result: Link keywords to knowledge_master_entries
    └─ Create thought_task_nodes + thought_edges
    ↓
  Return AI Response to UI

WRITING STYLE INJECTION (6 Endpoints)
================================================================================

Implementation:
  getUserWritingStyle(userId: string, channel?: string): Promise<string>
    ↓
  Query: SELECT body FROM inbox_messages
         WHERE direction='sent' AND channel=? (if specified)
         ORDER BY created_at DESC LIMIT 10
    ↓
  Filter: Min 5 chars, max 5 messages
    ↓
  Format: "## あなたの過去の送信メッセージ\n--- 送信例1 ---\n${message1}\n..."
    ↓
  Inject into System Prompt (before user message)

Applied to:
  ① generateReplyDraft (email/slack/chatwork)
  ② generateTaskChat (all phases)
  ③ Job structuring - schedule (greeting/closing)
  ④ Consultation reply (return email)
  ⑤ Not used: Keyword extraction (generic)
  ⑥ Not used: Knowledge clustering (generic)

Fallback: Empty string if no API key or <5 messages available

================================================================================
EMAIL SIGNATURE HANDLING
================================================================================

Storage: user_metadata.email_signature (Supabase auth)

Retrieval: getServerUserEmailSignature() → user_metadata.email_signature

Injection Points:
  ① Reply draft generation
  ② Job structuring (schedule)
  ③ Consultation reply

Application Rules:
  - ONLY for email channel
  - NOT in system prompt
  - Appended to draft AFTER Claude returns response
  - Format: draft.trimEnd() + '\n\n' + emailSignature

Example:
  AI draft: "ご返信ありがとうございます。確認いたしました。"
  Code appends: "\n\n" + "太郎\nNodeMap Inc.\n+81-3-xxxx-xxxx"
  Final: "ご返信ありがとうございます。確認いたしました。\n\n太郎\nNodeMap Inc.\n..."

CONVERSATION HISTORY MANAGEMENT
================================================================================

Task AI Chat:
  Stored: task_conversations (role, content, phase, created_at, turn_id)
  Passed to Claude: Last 20 messages (full text)
  Retention: Indefinite (unless task deleted)
  Truncation: None (size limited by query LIMIT 20)

Secretary Chat:
  Stored: secretary_conversations (for context reference)
  Passed to Claude: NONE (single turn, no history)
  Retention: 30 days (soft, no explicit cleanup)
  Rationale: Fresh DB data (messages/tasks/jobs) substitutes for history

Replay Mode:
  Stored: task_conversations
  Passed to Claude: Last 50 messages + snapshots + nodes
  Retention: Until task deleted
  Purpose: Re-enact completed task thinking

SECURITY & DATA PRIVACY
================================================================================

✓ No passwords/secrets in system prompts
✓ No API keys exposed to Claude
✓ All queries filtered by user_id (multi-tenant safe)
✓ Service role key bypasses RLS but scoped to user_id
✓ Contact names/emails injected at request time (not hardcoded)
✓ Conversation history retained indefinitely (no TTL for task_conversations)

⚠ Risks:
  - Writing style exposes past message content
  - Contact notes may contain sensitive info (not masked)
  - Secretary_conversations stored indefinitely
  - No audit trail of system prompts sent to Claude

CONCLUSION
================================================================================

NodeMap uses Claude API across 18 distinct call points with varying context
complexity. The largest context consumer is Secretary Chat (2000 tokens max),
which pulls fresh data from 10+ tables and 2 external APIs per request.

User-specific context (writing style, email signature, contact info) is
injected at request time and appended to system prompts. No sensitive data
is hardcoded.

Token usage averages 25-40k per active user per day, with knowledge clustering
and task summary generation as peak consumers. All endpoints have fallback
responses (template or demo).

The architecture is sound for multi-tenant operation, but lacks explicit
data retention policies and audit logging.
