# NodeMap データフロー全体像

## このドキュメントの目的

NodeMapのデータがどこから入り、どう加工され、どこに出ていくかを正確に説明する。図解作成の参考資料として使用する。

---

## 1. データの入口（3つの流入経路）

NodeMapに情報が入ってくるルートは3つある。

### 経路A: 会議録（3つのソース）

会議の内容がNodeMapに入る方法は3種類ある。

- **A-1: Gemini会議メモ（自動）** — Google Meetで「メモを取る」をONにすると、会議終了後にGeminiがGoogle Docsを自動生成する。NodeMapの定期ジョブ（毎日1回）がGoogleカレンダーの過去48時間のイベントをスキャンし、Gemini会議メモを検出して自動取り込む。検出方法はカレンダーイベントの添付ファイル確認 → なければDriveでタイトル検索の2段階。テキスト解析はAI不要（コードベースのパーサー）。
- **A-2: MeetGeek Webhook（自動）** — MeetGeekが会議を録画・文字起こしし、会議終了時にWebhookでNodeMapに自動通知する。全文トランスクリプト・サマリー・ハイライトを取得して保存。こちらのテキスト解析はClaude AI（有料）を使う。
- **A-3: 手動登録** — プロジェクトの検討ツリータブから、ユーザーがテキストを直接入力して会議録を登録する。

3つのソースとも、取り込み後は同じ「共通パイプライン」で処理される（後述）。

### 経路B: チャネルメッセージ（自動）

SlackとChatworkのメッセージが定期ジョブ（Cron）で自動同期される。1チャネル＝1プロジェクトの原則で紐づけ。

- Slackのチャネルメッセージ → inbox_messages に保存
- Chatworkのルームメッセージ → inbox_messages に保存
- メール → 現在休眠中（設定で無効化）

チャネルメッセージからも、会議録と対称的に、タイムライン・検討ツリー・ナレッジが自動生成される。

### 経路C: チャネルボット（Slack/Chatworkからの指示）

SlackやChatworkで「@NodeMap タスク作って」のようにメンションすると、NodeMapのボットが応答する。タスク作成、状況確認、メニュー表示などに対応。ボットは情報の参照はするが、基本的に変更操作はNodeMap画面から行う原則。

---

## 2. 共通パイプライン（会議録の自動処理）

経路A（Gemini / MeetGeek / 手動）で会議録がNodeMapに入ると、以下の処理が自動的に連鎖実行される。

### ステップ1: 議事録保存
会議録テキストが `meeting_records` テーブルに保存される。

### ステップ2: AI解析（またはGeminiパーサー）
- Gemini会議メモの場合 → コードベースのパーサーが構造化テキストを解析（AI不要・無料・高速）
- MeetGeek/手動の場合 → Claude AIがテキストを解析（有料・高精度）

解析で以下が抽出される：
- **要約（サマリー）** → meeting_records.ai_summary に保存
- **トピック（議題）** → 検討ツリーのノードに変換
- **アクションアイテム** → タスク提案として保存（担当者ごとに集約、文脈付き）
- **決定事項** → decision_log（意思決定ログ）に記録
- **未確定事項** → open_issues（未確定事項トラッカー）に記録
- **キーワード** → knowledge_master_entries（ナレッジ）に自動抽出

### ステップ3: タイムラインに自動追加
会議イベントが `business_events` テーブルに登録され、プロジェクトのタイムライン画面に自動表示される。タイムラインは完全に読み取り専用で、手動でイベントを追加することはできない。

### ステップ4: 検討ツリー生成
AI解析で抽出されたトピックが、検討ツリーのノードとして自動生成される。既に似たタイトルのノードがあれば更新、なければ新規作成（重複防止あり）。検討ツリーは「何が議論され、どう決まったか」の意思決定の記録。

### ステップ5: タスク提案 → 人間の承認
AI解析で抽出されたアクションアイテムが `task_suggestions` に保存される。検討ツリータブの「タスク提案パネル」に表示され、ユーザーが以下を編集してから承認する：
- タイトル
- 担当者（プロジェクトメンバーから選択）
- 優先度（高/中/低）
- 期限

承認されると正式な `tasks` レコードになる。**タスクは自動では作られず、必ず人間の承認を経る**。

## 3. タスクの流れ

### タスクの作成元（3つ）
- **会議録から** → 上記のタスク提案パネルで承認
- **チャネルボットから** → Slack/Chatworkで「タスク作って」メンション → 自動作成（これは例外的に人間の承認なしで即時作成）
- **手動作成** → タスク画面やSecretaryから直接作成

### タスクの管理
タスクは統合カンバンボード（/tasks画面）で管理される。4列構成：未着手(todo) → 進行中(in_progress) → レビュー(review) → 完了(done)。プロジェクト選択 + 担当者フィルタ + 日付フィルタで絞り込み可能。

### タスクの外部同期（双方向）
タスク作成時に、元のチャネルにも通知される：
- **Slack** → Block Kitリッチカード（「完了」ボタン付き）をチャネルに投稿
- **Chatwork** → ネイティブタスクAPIでタスクを作成

完了時も双方向同期される：
- NodeMapで完了 → Slack/Chatworkのタスクも完了に更新
- Slack/Chatworkで完了 → NodeMapのタスクも完了に更新

---

## 4. チャネルメッセージの流れ（経路B）

Slack/Chatworkのメッセージは定期ジョブで `inbox_messages` に同期される。そこから以下が自動生成される：

- **タイムライン** → メッセージイベントとして `business_events` に追加
- **検討ツリー** → メッセージ内容から議題を抽出してノード生成
- **ナレッジ** → キーワード自動抽出

会議録（経路A）と対称的な構造になっている。

---

## 5. 定期配信（ボットからチャネルへ）

NodeMapからSlack/Chatworkへ自動的に情報が配信される：

- **月曜 09:00** → 今週のブリーフィング（未確定事項数 + 今週のタスク + 予定会議）
- **金曜 17:00** → 今週の成果レポート（完了タスク + 新規決定事項）
- **毎日 09:30** → アラート（期限超過タスク、滞留している未確定事項、MS期限接近）

社内/社外チャネルで配信内容が自動的にフィルタされる（社外には未確定事項を見せない等）。

---

## 6. 3つの常設データ

プロジェクトごとに、AI解析の精度を上げるための3つの常設データがある：

- **未確定事項（open_issues）** — 結論が出なかった事項を追跡。21日以上放置で「stale」に自動変更。AI解析で解決が検知されると自動クローズ。
- **意思決定ログ（decision_log）** — 「決まったこと」の不変記録。変更時は新レコード作成+旧レコードを「superseded」に。チェーンで辿れる。
- **会議アジェンダ（meeting_agenda）** — 上記2つ + タスク進捗から翌営業日のアジェンダを自動生成。

AI解析時に、これらの過去データがプロンプトに注入され、文脈を踏まえた解析ができる。

---

## 7. 階層構造

NodeMapのデータは5階層で整理される：

```
組織（Organization）
└── プロジェクト（Project）
    ├── テーマ（Theme）※任意の中間レイヤー
    │   └── マイルストーン（Milestone）
    │       └── タスク（Task）
    └── ジョブ（Job）※定型業務・やることメモ
```

- **タスク** = 思考を伴う作業。マイルストーン配下に配置必須。
- **ジョブ** = 定型業務やメモ。プロジェクト直下に配置。SEOレポートや定例MTGなどの定期的な作業に使う。

---

## 8. 全体の流れを一文で

「Google Meet / MeetGeek / 手動で会議録がNodeMapに入ると、AI解析で要約・トピック・タスク候補・決定事項・未確定事項が自動抽出され、タイムライン・検討ツリー・ナレッジに自動反映される。タスクは人間が承認してから正式に作成され、Slack/Chatworkと双方向で同期される。チャネルメッセージも定期ジョブで取り込まれ、同じパイプラインで処理される。」
