# NodeMap 改善計画（全機能）

作成日: 2026-03-05

> このドキュメントは全機能の改善方針をまとめたもの。
> 実装順序: **共通ルール → 機能単位で順次実装** の原則に従う。

---

## 0. 共通ルール（全機能に影響）

### 0-1. 営業時間ルール

| 項目 | ルール |
|---|---|
| 営業日 | 平日のみ（土日祝は除外） |
| 営業時間 | 10:00〜19:00 |
| AI出力制限 | 10:00より前、19:00以降の時間帯を候補に出すのはNG |
| ユーザー裁量 | 出力後にユーザーが9:00に変更する等は自由 |

**適用箇所**: `findFreeSlots()` / 秘書AI日程調整 / インボックス返信下書き（scheduleMode）/ タスクのカレンダー登録提案

### 0-2. カレンダー命名ルール・スキップルール

| 予定の種類 | 命名規則 | 空き検索時 |
|---|---|---|
| NodeMapタスク予定 | `[NM-Task] タスク名` | **スキップ（空きとみなす）** |
| NodeMapジョブ予定 | `[NM-Job] ジョブ名` | **スキップ（空きとみなす）** |
| ミーティング・外部予定 | 上記プレフィックスなし | **ブロック（空きではない）** |
| ユーザー手動登録 | ルールなし | **ブロック（空きではない）** |

**実装方針**: `findFreeSlots()` で `extendedProperties` または予定タイトルの `[NM-Task]` / `[NM-Job]` プレフィックスをチェックし、該当する予定をスキップして計算。

**将来検討**: NodeMap専用Googleカレンダーアカウントの導入（タスク予定を別カレンダーに分離）

### 0-3. 1チャンネル＝1プロジェクトの原則

| 項目 | ルール |
|---|---|
| 基本方針 | 1つのチャットグループ/チャンネル = 1つのプロジェクト |
| 効果 | ビジネスログの要約精度向上、タスク範囲の明確化、チャネル紐づけの自動化 |
| 例外 | メール・LINEなど1:1のやり取りは手動紐づけ |

### 0-4. AI応答の思考プリセット（伸二メソッド）

全AIの応答に以下の思考フレームワークを注入する:

**思考哲学**:
- 表面的な正解よりも「意味のある選択」を重視
- 一般論ではなく「構造理解」を重視
- 答えよりも「思考プロセス」を重視

**思考エンジン**:

1. **階層思考（Vertical Thinking）** — 「なぜ」を繰り返し本質に近づく
   - Layer1: 事象 → Layer2: 理由 → Layer3: 構造 → Layer4: 心理 → Layer5: 原理
   - 回答では必ず1〜2段階深いレイヤーまで潜行する

2. **思考の飛び地（Horizontal Jump）** — 他分野・アナロジー・異業界・心理学で拡張
   - 階層思考だけでなく横方向の連想で可能性を提示

3. **深さと飛び地の往復** — 潜る（Why）→ 飛ぶ（連想）→ もう一度潜る
   - この運動を繰り返して洞察を生み出す

**ストーリー設計**: 課題の構造 → 顧客の心理 → 解決の意味 → 提案の価値

**対話スタイル**: 壁打ち型。よく使う表現: 「そもそも」「客観的に見ると」「根本的には」「構造で見ると」

**伸二レビュー（最終チェック）**:
1. 構造理解があるか
2. 課題の深さは十分か
3. 顧客心理が含まれているか
4. 提案がストーリーになっているか

**カスタマイズ**: ユーザープロフィールの性格タイプを参照し、哲学的表現の濃淡を調整。

**適用箇所**: タスクAI会話 / 秘書チャット / 返信下書き（ビジネス文脈がある場合）

## 1. インボックス

### 1-1. メール機能の休眠化

| 項目 | 内容 |
|---|---|
| 目的 | 社内連絡がChatworkメインのため、メール取得の優先度を下げる |
| 方針 | ソースコードは残す。処理・表示を無効化（フラグ制御） |
| 実装案 | 環境変数 or 設定画面で `EMAIL_ENABLED=false` → fetchDiff / 表示をスキップ |
| 復帰方法 | フラグをtrueに戻すだけで復帰 |

### 1-2. インボックスのリアルタイム更新

| 項目 | 内容 |
|---|---|
| 現状 | 更新ボタンを押さないと最新化されない |
| 改善案 | ページ表示時に自動でバックグラウンド取得を開始。ポーリング間隔（例: 3分）で差分チェック |
| 代替案 | 最低限、ページ遷移で戻ってきたときに自動更新 |

### 1-3. Chatwork/Slackの返信下書き対応

| 項目 | 内容 |
|---|---|
| 現状 | メールにのみ返信下書きを表示 |
| 改善 | Chatwork / Slack メッセージにも返信下書きボタンを表示 |
| 注意 | チャネル別のトーン調整（メール=丁寧 / Slack=カジュアル / CW=標準）は既存 |

### 1-4. 過去のやり取り変遷パネル

| 項目 | 内容 |
|---|---|
| 目的 | 受信メッセージの右側に、その相手との過去のやり取り履歴を表示 |
| UI | メッセージ詳細画面の右カラムに新設 |
| データソース | `inbox_messages` を `from_address` でグルーピング、時系列表示 |
| 蓄積方式 | バックグラウンドでDB蓄積（過去分）。最新1回前までの変遷を表示 |
| 注意書き | 「最新の受信は反映されていない場合があります」を表示 |
| AI活用 | この変遷をAIコンテキストに注入 → 返信下書きの品質向上 |

## 2. カレンダー

### 2-1. 営業時間の適用

→ 共通ルール 0-1 参照。`findFreeSlots()` のパラメータを変更:
- `dayStartMs` = 10:00（現行9:00から変更）
- `dayEndMs` = 19:00（現行18:00から変更）
- 土日祝を除外（祝日判定の追加が必要）

### 2-2. タスク予定スキップ

→ 共通ルール 0-2 参照。`findFreeSlots()` 内で:
1. Google Calendar APIからイベント取得
2. タイトルが `[NM-Task]` または `[NM-Job]` で始まるイベントを除外
3. 残りのイベントで空き時間を計算

### 2-3. 複数候補の出力

| 項目 | 内容 |
|---|---|
| 現状 | 空き時間を最大20スロット返す |
| 改善 | 同じ日付内に複数の空きがあれば全て出す。ビジネスマナーとして候補は多く |
| ルール | 秘書経由でもインボックス経由でも同一ロジック（`findFreeSlots()` を共通利用） |

## 5. 組織・コンタクト・ビジネスログ

### 5-1. UI階層の統合

| 項目 | 内容 |
|---|---|
| 現状 | 組織ページ・プロジェクトページ・ビジネスログページがバラバラ |
| 改善 | 組織詳細 → プロジェクト一覧 → タスク/ドキュメント/ビジネスログを1画面で操作 |
| 階層 | 組織 > プロジェクト > タスク ｜ ドキュメント ｜ ビジネスログ（並列） |

### 5-2. 1チャンネル＝1プロジェクトの自動紐づけ

→ 共通ルール 0-3 参照。

| 項目 | 内容 |
|---|---|
| 実装 | チャンネル受信時に `project_channels` を自動検索 → 一致すればプロジェクトに自動紐づけ |
| 新規 | 未登録チャンネルは「プロジェクト未紐づけ」として通知 → ユーザーが紐づけ |
| 効果 | コンタクト自動生成 → チャンネル経由でプロジェクト自動特定 → ビジネスログ自動蓄積 |

### 5-3. ビジネスログの位置づけ明確化

| 項目 | 内容 |
|---|---|
| 方針 | ビジネスログは「自動蓄積される振り返り・共有の場」 |
| 操作 | 基本的にユーザーは何もしない。例外: 手動メモ追加、ファイルアップロード |
| 表示 | プロジェクト詳細画面のタブとして表示（独立ページではなく） |

## 7. 秘書AI

### 7-1. 大原則: 秘書ができないことをなくす

| 項目 | 内容 |
|---|---|
| 最終目標 | 将来的にUIを秘書チャットのみに集約する |
| 現状 | 各画面（タスク・コンタクト・組織等）で個別操作が必要な機能がある |
| 方針 | **全ての機能を秘書経由で実行可能にする**。各ページは利便性のための補助UI |
| 実装ルール | 今後の全改善項目は「秘書からも同じことができる」前提で作る |

### 7-2. 秘書に接続すべき機能（現状の不足）

秘書から実行できないものを洗い出し、全てintent + カードアクションで対応する:

| 機能 | 現状 | 対応 |
|---|---|---|
| コンタクト作成 | ○ 対応済み | — |
| 組織作成 | ○ 対応済み | — |
| プロジェクト作成 | ○ 対応済み | — |
| タスク作成 | ○ 対応済み | — |
| チャンネル紐づけ | × 手動のみ | intent追加: 「このチャンネルを○○プロジェクトに紐づけて」 |
| ファイルアップロード | △ Drive保存のみ | タスクへの外部資料取込も秘書経由で |
| ビジネスログ閲覧 | △ 要約のみ | プロジェクト別の詳細閲覧も対応 |
| ナレッジ確認/承認 | △ 提案確認のみ | 期間別ノード表示も秘書から |
| 設定変更 | × 不可 | メール休眠ON/OFF等の設定も秘書から |
| 組織→PJ→タスク階層ナビ | × 不可 | 「○○組織のプロジェクト一覧」等のintent追加 |

### 7-3. 秘書UIの段階的進化

| フェーズ | 内容 |
|---|---|
| 現在 | 秘書 + 各ページ併用 |
| 中期 | 秘書で全操作可能。各ページは閲覧・一括操作用 |
| 最終 | 秘書UIのみ。各ページは設定・管理用に縮小 |

## 8. ツール全体の設計思想

### 8-1. NodeMapの本質的な価値

| 項目 | 内容 |
|---|---|
| 最重要機能 | **思考マップ** — ノードが勝手に溜まる「思考の貯蔵箱」 |
| コミュニケーション | Slack/Chatwork/メールなど好きな場所でやる。NodeMapはそれを自動同期・自動蓄積 |
| タスク進行 | **ここだけはNodeMap内でやるべき** — AI壁打ち・フェーズ管理・思考ノード生成のため |
| 各ページの役割 | 一応全機能は操作可能にするが、最終的にはコミュニケーション管理はSlack/CWの方が便利 |

### 8-2. 連携の大原則

```
外部（Slack/CW/Gmail/Calendar）
  ↓ 自動同期
NodeMap（自動蓄積・自動分類・自動ノード化）
  ↓ 思考マップに集約
ユーザーの知的資産
```

- **勝手に同期**: ユーザーが意識しなくても外部サービスからデータが流れ込む
- **勝手に蓄積**: メッセージ→ビジネスイベント→ナレッジノードが自動生成
- **タスクだけはNodeMapで**: AI壁打ちと思考マップ生成のため、タスク進行はここで行う

## 実装優先順位（提案）

| 順番 | 内容 | 理由 |
|---|---|---|
| **Phase A** | 共通ルール（営業時間・命名・1Ch=1PJ・思考プリセット） | 全機能の土台。先に固めないと各機能で矛盾が生じる |
| **Phase B** | インボックス改善（メール休眠・リアルタイム更新・CW/Slack返信・変遷パネル） | 日常最も使う画面。体感改善が大きい |
| **Phase C** | カレンダー改善（営業時間適用・タスクスキップ・複数候補） | 共通ルール適用の実装。秘書AIにも波及 |
| **Phase D** | 組織・ビジネスログUI統合（階層統合・自動紐づけ） | 構造変更は早めに。後の機能に影響 |
| **Phase E** | タスクAI会話改善（思考プリセット・外部資料取込） | AI品質向上。共通ルール0-4を実際に適用 |
| **Phase F** | ナレッジ可視化（期間別表示） | 比較的小さい変更 |
| **Phase G** | 秘書AI全機能接続（不足intentの追加・カードアクション拡張） | 全Phase A〜Fの実装後に、秘書から全操作可能にする仕上げ |

### 実装の大原則

> **全ての改善項目は「秘書からも同じことができる」前提で作る。**
> Phase A〜Fの各実装時に、対応するintent/カードアクションを同時に作成する。
> Phase Gは「漏れの回収」として、秘書から実行できない機能がないことを最終確認する。
