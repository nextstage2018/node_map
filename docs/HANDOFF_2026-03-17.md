### エラー1: タスク画面の担当者フィルタ ― 残課題あり

**元の問題**: 「全員」フィルタで他メンバーのタスクが表示されない

**修正済み**:
- `src/app/api/tasks/my/route.ts` に `show_all=true` パラメータ追加（user_idフィルタをスキップ）
- `src/app/tasks/page.tsx` で「全員」選択時に `show_all=true` を送信
- `src/components/v4/TeamTaskCard.tsx` にプロジェクト名表示＋担当者常時表示を追加

**残課題**: 全員表示にしても全カードに「自分」と表示される（矛盾）

**原因分析**:
- `assigned_contact_id` が未設定のタスクが多い（Slack/CWからの自動作成タスクでは設定されるが、手動作成タスクでは未設定）
- APIの `enrichedTasks` で `assignee_name` は `assigned_contact_id` → `contact_persons.name` で解決
- `assigned_contact_id` が null → `assignee_name` も null → TeamTaskCardで「自分」フォールバック表示
- **本質**: `user_id`（タスク作成者）→ ユーザー名への名前解決が必要

**修正方針**:
`src/app/api/tasks/my/route.ts` の `enrichedTasks` マッピングで、`assigned_contact_id` がnullの場合に `user_id` → ユーザー表示名を解決するフォールバックを追加する。方法としては:
1. `user_id` の一意リストを収集
2. Supabaseの `auth.users` または `user_service_tokens` 等からユーザー名を取得（または `contact_persons.linked_user_id` の逆引き）
3. `assignee_name` が null の場合に `user_id` ベースの名前をセット

**関連ファイル**:
- `src/app/api/tasks/my/route.ts` — タスク一覧API（125-131行: enrichedTasks マッピング）
- `src/app/tasks/page.tsx` — カンバンページ（239-252行: filteredTasks フィルタロジック）
- `src/components/v4/TeamTaskCard.tsx` — タスクカード（117-120行: 担当者表示）

**残課題1: アップロード資料が「受領資料」タブに表示される**

**原因**:
`src/services/drive/driveClient.service.ts` の `recordDocument()` 関数で `direction` のデフォルト値が `'received'` になっている（1712-1715行付近）。手動アップロード時に `direction` パラメータが渡されないため、デフォルトの `'received'` が適用 → 「受領資料」カテゴリに分類される。

カテゴリ判定ロジック（`getDocuments` 関数内）:
```
received: source_channel IS NOT NULL OR direction = 'received'
registered: source_channel IS NULL AND direction != 'received'
```

**修正方針**:
2つの選択肢:
- **案A（推奨）**: `recordDocument()` のデフォルトdirectionを `'submitted'` に変更。ただし他の呼び出し元（メール添付自動取り込み等）への影響を確認する必要あり
- **案B**: `POST /api/drive/documents` のファイルアップロード時に `direction: 'submitted'` を明示的に渡す
- URL登録（direct insert）は `direction` カラムを設定していないため `null` → registered に正しく分類される

**残課題2: フォルダ格納ルール見直し**

ユーザーから「フォルダの格納ルールから見直したい」との要望あり。
現在の構造:

```
[NodeMap] 組織名/
└── プロジェクト名/
    ├── 定期イベント/       ← jobId + jobName 指定時
    ├── 会議議事録/         ← meetgeek/gemini 由来
    └── マイルストーン/
        └── MS名/
            └── タスク名/   ← taskId + taskName + milestoneId 指定時
```

今回の修正で `jobId` を送らなくなったため、定期イベントフォルダへの振り分けが機能しなくなっている。
また `drive_documents.job_id` のFK制約が `jobs` テーブル参照のため、`project_recurring_rules` のIDを格納できない。

**検討ポイント**:
- `drive_documents.job_id` FK制約をどうするか（ALTER TABLE で参照先変更 or 新カラム追加 or 廃止）
- 定期イベントへの資料紐づけが必要か、タグだけで十分か
- ユーザーがどういう格納構造を望んでいるか（次スレッドで確認すべき）
