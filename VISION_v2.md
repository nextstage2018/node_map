## 1. NodeMapとは何か

NodeMapは「秘書に話しかけるだけで、仕事が進み、記録が溜まる」ツールである。

従来のビジネスツール: 人間が画面を開き、フォームに入力し、ダッシュボードを読み解き、タスクを管理する。
NodeMap: AIが探し、書き、分析し、実行する。人間は確認と判断だけする。

### 1.1 解決する課題

- 属人的な業務知識が共有されない（担当者の頭の中にしかない）
- コミュニケーションチャネルが分散し、情報が散逸する
- 簡易作業に時間を取られ、本質的な仕事に集中できない
- チームメンバーの思考プロセスが見えず、品質にばらつきが出る
- 色んな画面で色んな情報を登録するから、抜け漏れが起き利便性も出しづらい

### 1.2 コア原則

1. **会話ファースト**: ユーザーの主要インターフェースは「秘書AIとの会話」。画面遷移は最小限
2. **自動蓄積**: 使っているだけでビジネスログ・ナレッジ・思考プロセスが溜まる
3. **AI代行**: 定型的・簡易な作業はAIが自律実行し、人は確認するだけ
4. **思考の可視化**: 重要な仕事でのAI対話プロセスが、チーム全体の参考資料になる
5. **個人最適化**: 使い込むほど秘書AIが本人に近づき、代行の精度が上がる

### 1.3 従来型UI vs AIネイティブUI

| 従来型（やめたい） | AIネイティブ（目指す姿） |
|---|---|
| 人間が検索する | AIが探して、人間は選ぶだけ |
| 人間がフォームに入力する | AIが書いて、人間は確認するだけ |
| 人間がダッシュボードを読み解く | AIが分析と提案をして、人間は承認するだけ |
| 人間がタスクを管理する | AIが実行して、人間は評価するだけ |
| 画面を切り替えて操作する | 1つの会話で指示するだけ |

## 2. 全体アーキテクチャ

### 2.1 UIの根本思想: 秘書AIが唯一の操作面

```
┌──────────────────────────────────────────┐
│                                          │
│            秘書AI（会話画面）              │
│                                          │
│   ユーザーが話しかける唯一のインターフェース  │
│                                          │
│   「メール見せて」→ メッセージ一覧を表示      │
│   「これ日程調整しといて」→ ジョブ実行        │
│   「提案書の続きやろう」→ タスク対話開始      │
│   「○○社の最近の動きは？」→ ログ検索        │
│   「田中さんの思考マップ見せて」→ 可視化表示  │
│                                          │
└───────────────┬──────────────────────────┘
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 会話内カード  インライン表示  別画面展開
 (承認/選択)   (一覧/詳細)   (思考マップ等)
```

### 2.2 画面の3層構造

**第1層: 秘書AI会話（メイン画面）**
- ユーザーが常にいる場所。全操作の起点。
- AIが朝の報告、メッセージ要約、ジョブ進捗、タスクリマインドを会話形式で提示
- ユーザーは自然言語で指示するだけ

**第2層: インラインカード（会話内に展開）**
- メッセージ一覧、ジョブの承認カード、タスク進捗カードなど
- 会話画面の中にリッチなUIパーツとして表示される
- ボタン操作（承認/却下/修正）も会話内で完結

**第3層: ビジュアル画面（必要な時だけ開く）**
- 思考マップ（Canvas描画が必要）
- ビジネスログの時系列ブラウズ
- 設定画面
- 秘書AIの会話から「〇〇を開いて」でシームレスに遷移

### 2.3 情報の流れ

```
外部チャネル（メール/Slack/Chatwork/LINE...）
        │
        ▼
  ┌─────────────┐
  │  統合インボックス  │  ← バックエンドで受信（画面ではない）
  └──────┬──────┘
         │
    秘書AIが自動トリアージ
         │
    ┌────┼────┐
    ▼    ▼    ▼
  ジョブ  タスク  ログのみ
  (AI自律) (人+AI)  (自動記録)
    │    │    │
    ▼    ▼    ▼
  ┌─────────────┐
  │  ビジネスログ   │  ← すべての処理結果が自動蓄積（バックエンド）
  └──────┬──────┘
         │
    ┌────┴────┐
    ▼         ▼
  ナレッジDB   思考マップ
  (組織共有)   (個人の脳の可視化)
```

### 2.4 機能の6本柱（再定義）

| # | 柱 | 概要 | UIとしての表れ方 |
|---|---|---|---|
| 1 | 秘書AI | すべての操作の起点。個人最適化されたAI | メイン画面（会話） |
| 2 | 統合インボックス | 多チャネルの受信を一元管理 | 秘書が会話内で報告・表示 |
| 3 | ジョブ（AI代行） | 簡易作業をAIが自律実行する | 秘書が下書き→承認カード表示 |
| 4 | タスク（AI協業） | 重要作業を人+AIで進め、思考を記録 | 秘書との深い対話モード |
| 5 | ビジネスログ | 全活動が自動蓄積される組織の記録 | 必要時にビジュアル画面で閲覧 |
| 6 | 思考マップ＋ナレッジ | 思考プロセスの可視化と組織の知識基盤 | 必要時にビジュアル画面で閲覧 |

**重要な転換**: インボックス・ジョブ・タスクは「独立した画面」ではなく、秘書AIの会話内で処理される。

## 4. ジョブ（AI代行）の詳細設計

**名称検討**: 「ジョブ」→「おまかせ」「代行」「オートアクション」なども候補

### 4.1 ジョブの種別

| 種別 | 具体例 | AIの処理 |
|---|---|---|
| 日程調整 | 「来週空いてる日を3つ提案して」 | カレンダー参照→候補生成→送信文作成 |
| 確認＆返信 | 「納品日を先方に確認して」 | 確認文生成→承認後送信→返信待ち |
| 返信 | 「お礼を返しておいて」 | コンタクト情報参照→文体合わせて返信文生成 |
| 定型作業 | 「毎朝の報告メールを送って」 | テンプレート＋前日データ→自動送信 |

### 4.2 ジョブのライフサイクル

```
会話で指示 → AI下書き生成 → 会話内で承認カード表示
                                    │
                          ┌────────┼────────┐
                          ▼        ▼        ▼
                        承認     修正      却下
                          │        │
                          ▼        ▼
                      AI実行   再下書き→再承認
                          │
                          ▼
                    完了（ログ自動記録）
```

### 4.3 ジョブのステータス

| ステータス | 意味 |
|---|---|
| draft | AI下書き生成済み、人の確認待ち |
| approved | 人が承認済み、AI実行待ち |
| executing | AI実行中 |
| done | 完了 |
| failed | 実行失敗（リトライ可能） |

## 5. タスク（AI協業）の詳細設計

**名称検討**: 「タスク」→「プロジェクトワーク」「ワーク」なども候補

### 5.1 タスクの本質

タスクは「人間が考えて、AIがそれを支える」もの。ジョブとの最大の違いは、思考マップに記録されること。チームの知的資産になる。

### 5.2 種の廃止

現行の「種（Seed）」は廃止し、タスクに統合する。

- 旧: インボックス → 種ボックス → AI会話 → タスク化
- 新: インボックス → 秘書に「タスクにして」→ AI構造化 → タスク登録

タスク作成時にAIが自動的にゴール/内容/優先度/期限を構造化するため、中間段階は不要。

### 5.3 タスクのフェーズ

```
構想（ideation）→ 進行（progress）→ 結果（result）
```

- **構想**: 「何をどう進めるか」をAIと対話して明確にする
- **進行**: 実際の作業をAIと対話しながら進める
- **結果**: 成果を整理し、ナレッジを確定する

### 5.4 タスクの種別

| 種別 | 具体例 | チーム共有 |
|---|---|---|
| 資料作成 | 提案書、報告書、プレゼン | する |
| ビジネス設計 | 新規事業計画、戦略立案 | する |
| 提案作業 | 見積もり、提案 | する |
| その他重要作業 | 属人的だがナレッジ化すべき業務 | する |

### 5.5 会話画面でのタスク対話

秘書AIとの通常会話から「タスクモード」にシームレスに切り替わる。

- 通常会話: 秘書として指示を受ける
- タスクモード: 伴走パートナーとして一緒に考える

切り替えのトリガー:
- 「提案書の続きやろう」→ タスクモードに入る
- 「ありがとう、一旦ここまで」→ 通常会話に戻る
- タスクモード中の会話はすべて思考マップに記録される

## 6. ビジネスログ（自動蓄積）

### 6.1 目的

あらゆる業務活動が自動的に蓄積される、組織の活動記録。「勝手に溜まる」ことが最も重要。

### 6.2 自動蓄積トリガー

| ソース | 記録内容 | トリガー |
|---|---|---|
| インボックス | 送受信メッセージ | メッセージの送受信時 |
| ジョブ | AI代行の実行記録 | ジョブ完了時 |
| タスク | タスクの進捗・完了記録 | フェーズ遷移時 |
| コンタクト | やり取り履歴 | メッセージ送受信時 |

### 6.3 ビジネスログの構造

```
組織
  └─ プロジェクト
       └─ チャネル紐づけ（Slackチャンネル、CWルーム等）
       └─ 活動ログ（自動蓄積）
            ├─ メッセージ（チャネル経由で自動分類）
            ├─ ジョブ完了記録
            ├─ タスク進捗記録
            └─ 手動メモ・議事録
```

### 6.4 UIでの表れ方

ビジネスログは基本的に**秘書AIに聞く**ことで参照する。

```
ユーザー: ○○社の最近の動きは？

秘書: ○○社の直近1週間のサマリーです。
  ・Webリニューアル案件: 見積もり依頼あり（鈴木様）
  ・広告運用: 月次レポート配信済み
  ・新規相談: なし
  → [詳細をビジネスログで見る]
```

時系列ブラウズや組織横断での俯瞰が必要な場合のみ、ビジュアル画面（/business-log）を開く。

## 7. 思考マップ＋ナレッジ

### 7.1 ナレッジDB（組織共有の知識基盤）

- マスターデータ（knowledge_master_entries）は全ユーザー共有
- 同じキーワードは1レコード（重複なし）
- 個人との紐づけ（thought_task_nodes）は個人ごと

```
【ナレッジDB（共有）】
  キーワード「LTV」← 全員で1レコード
  使用者: 田中, 鈴木, 佐藤 ← 紐づけは個人ごと

【個人の思考マップ】
  田中の思考: LTV → CAC → ROAS → ...
  鈴木の思考: LTV → チャーン率 → ...
```

ナレッジDB自体の管理画面は管理者向けのみ。一般ユーザーはAIとの対話や思考マップを通じて間接的に参照する。

### 7.2 思考マップ（個人の脳の可視化）

**目的**: チームメンバーが「あの人がどう考えたか」を参照できるようにする。

**閲覧モード**:
1. **全体マップ**: ある人の全タスクにわたるナレッジノードの全体地図
2. **個別トレース**: 特定タスクでの思考の流れ
3. **比較**: 2人の思考動線を重ねて表示
4. **リプレイ**: 完了タスクの思考を再現し、当時の思考についてAIに質問

思考マップはCanvas描画が必要なため、ビジュアル画面（/thought-map）として維持する。ただし、秘書AIから「思考マップを開いて」で遷移できる。

## 8. コンタクト・組織

### 8.1 設計方針

- 組織（会社）→ コンタクト（個人）の階層構造
- 全ユーザーで共有（同じ会社を二重登録しない）
- インボックスのメッセージから自動検出・自動提案
- プロジェクトとの紐づけで、ビジネスログの自動整理に寄与

### 8.2 UIでの表れ方

コンタクト・組織の登録や参照も秘書AIとの会話で行える。

```
ユーザー: 鈴木さんの情報教えて

秘書: 鈴木一郎様の情報です。
  所属: ○○株式会社 営業部
  連絡先: suzuki@example.com / Slack
  直近のやり取り: 3/1 見積もり依頼
  進行中プロジェクト: Webリニューアル
  → [コンタクト詳細を開く]
```

組織一覧やコンタクト一覧のブラウズが必要な場合のみ、ビジュアル画面を開く。

## 9. 秘書AIのパーソナライズ

### 9.1 学習するデータ

| カテゴリ | データソース | 学習内容 |
|---|---|---|
| 文体・トーン | 過去の送信メッセージ | チャネル別の書き方、敬語、口調 |
| 判断基準 | ジョブの承認/修正パターン | 何を重要と判断するか、優先順位 |
| 知識・専門性 | タスクのAI対話履歴 | 本人の専門分野、思考パターン |
| 人間関係 | コンタクト情報＋やり取り履歴 | 誰とどう接するか、関係性の濃淡 |

### 9.2 パーソナライズの段階

```
Lv.1（初期）:   汎用AI。一般的な文体で返信生成
Lv.2（1ヶ月）:  送信履歴から文体の傾向を学習
Lv.3（3ヶ月）:  ジョブ修正パターンから判断基準を学習
Lv.4（6ヶ月）:  コンタクトごとの対応パターンを把握
Lv.5（1年〜）:  本人の思考パターンまで再現可能に近づく
```

## 11. データベース設計の方針

### 11.1 テーブル整理

| テーブル | 方針 |
|---|---|
| seeds | 段階的廃止。既存データはタスクに移行 |
| seed_conversations | タスクに移行後は不要 |
| jobs | status拡張（draft/approved/executing/done/failed）。カラム追加 |
| tasks | seed_id依存解消。task_type維持（personal/group） |
| idea_memos | 秘書会話に統合。テーブル自体は維持（secretary_memos にリネーム検討） |
| knowledge_master_entries | 組織共有を明確化（user_idなし） |
| thought_task_nodes | user_id（個人の思考）は維持 |
| business_events | type拡張。自動生成トリガー追加 |
| secretary_context | **新設**: 秘書AIのパーソナライズデータ |

### 11.2 秘書AIのコンテキストテーブル（新設案）

```sql
-- 秘書AIが学習したパーソナライズデータ
CREATE TABLE IF NOT EXISTS secretary_context (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id TEXT NOT NULL,
  context_type TEXT NOT NULL,    -- 'writing_style' | 'judgment' | 'preference'
  context_key TEXT NOT NULL,     -- 例: 'email_tone', 'priority_criteria'
  context_value JSONB NOT NULL,  -- 学習した内容（構造化データ）
  source_count INT DEFAULT 1,    -- 学習元のデータ数
  confidence FLOAT DEFAULT 0.5,  -- 確信度
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

### 11.3 ジョブの自律実行に必要な拡張

```sql
ALTER TABLE jobs ADD COLUMN IF NOT EXISTS approved_at TIMESTAMPTZ;
ALTER TABLE jobs ADD COLUMN IF NOT EXISTS executed_at TIMESTAMPTZ;
ALTER TABLE jobs ADD COLUMN IF NOT EXISTS execution_log TEXT;
ALTER TABLE jobs ADD COLUMN IF NOT EXISTS reply_to_message_id TEXT;
ALTER TABLE jobs ADD COLUMN IF NOT EXISTS target_contact_id TEXT;

-- statusのCHECK制約更新
-- 'draft' | 'approved' | 'executing' | 'done' | 'failed'
```
